## 2. 테넌트 Bypass 지원

### 배경

현재 `@BypassTenancy()` 데코레이터는 HTTP 가드만 우회한다. 로드맵에서는 "Prisma 레벨 bypass"를 이월 항목으로 명시했으나, 실제로 분석하면:

**HTTP 엔드포인트의 Prisma bypass는 이미 동작한다.** `@BypassTenancy()` 라우트에서 테넌트 헤더가 없으면 미들웨어가 `tenantId = null`로 통과시키고, Prisma 확장의 `$allOperations`는 `!tenantId`일 때 `query(args)`를 직접 호출한다 (prisma-tenancy.extension.ts:77). 즉, HTTP bypass + Prisma bypass가 이미 연동된다.

**부족한 것은 프로그래밍 bypass다.** 백그라운드 작업, 크론잡, 관리자 로직 등 HTTP 요청 밖에서 테넌트 격리를 의도적으로 우회하는 API가 없다.

### 리서치 기반 사용 시나리오 (빈도순)

1. **관리자 대시보드** — super-admin이 모든 테넌트 데이터 조회
2. **백그라운드 작업** — 크론잡, 큐 워커 (HTTP 요청 없이 실행)
3. **크로스테넌트 리포팅** — 전체 매출 집계, 사용량 메트릭
4. **마이그레이션/시딩** — DB 시딩, 데이터 마이그레이션
5. **유니크 검증** — 전체 테넌트 대상 이메일 중복 체크
6. **테넌트 룩업** — 로그인 시 테넌트 결정 (부트스트랩)

### 레퍼런스 (타 생태계)

- Rails `acts_as_tenant`: `ActsAsTenant.without_tenant { ... }` 블록
- Python `django-scopes`: `scopes_disabled()` 컨텍스트 매니저
- .NET EF Core: `.IgnoreQueryFilters()` 쿼리 메서드
- Laravel: `.withoutTenancy()` 쿼리 메서드

### 설계

**API: 블록/컨텍스트 패턴 (가장 인기 있는 패턴)**

```typescript
// 프로그래밍 bypass — 백그라운드 작업, 관리자 로직, 크론잡
const allUsers = await tenancyService.withoutTenant(async () => {
  return prisma.user.findMany(); // RLS 미적용, 모든 테넌트 데이터
});

// HTTP 엔드포인트 bypass — 기존과 동일 (이미 동작함)
@Get('/admin/users')
@BypassTenancy()
async getAllUsers() {
  return this.prisma.user.findMany(); // 가드 우회 + Prisma 자동 bypass
}
```

**구현 전략:**

1. `TenancyContext`에 `runWithoutTenant()` 메서드 추가:
   - `AsyncLocalStorage.run()`을 `tenantId`가 없는 빈 스토어 또는 `null` 센티넬로 호출
   - 기존 `TenantStore` 인터페이스 변경 없음 — `run()`의 오버로드가 아닌 별도 메서드
   - 내부 구현: `storage.run({ tenantId: null as unknown as string }, callback)`
   - `getTenantId()`는 `tenantId`가 `null`이면 `null` 반환 (현재와 동일)

2. `TenancyService`에 `withoutTenant()` 공개 메서드 노출:
   ```typescript
   async withoutTenant<T>(callback: () => T | Promise<T>): Promise<T> {
     return this.context.runWithoutTenant(callback);
   }
   ```

3. Prisma 확장 변경 없음:
   - `tenantId === null`이면 이미 `query(args)` 직접 호출 (현재 동작)
   - `withoutTenant()`가 context를 null로 설정하면 자연스럽게 bypass

4. 에러 전파: `AsyncLocalStorage.run()`은 콜백의 예외를 그대로 전파한다. `withoutTenant()`도 동일하게 동작하며, context 상태는 `run()` 종료 시 자동 복원된다.

**핵심: `tenantId === null`이 곧 bypass.** 새 플래그나 상태를 추가하지 않는다.

### 파일 변경

| 파일 | 변경 |
|------|------|
| `src/services/tenancy-context.ts` | `runWithoutTenant()` 메서드 추가 |
| `src/services/tenancy.service.ts` | `withoutTenant()` 공개 메서드 추가 |
| `test/tenancy-context.spec.ts` | `runWithoutTenant()` 동작, 에러 전파, 중첩 테스트 |
| `test/tenancy.service.spec.ts` | `withoutTenant()` 테스트 |
| `test/e2e/prisma-tenancy.e2e-spec.ts` | 실제 PostgreSQL에서 bypass 시 전체 데이터 조회 검증 |
| `README.md` | `@BypassTenancy()` + Prisma 연동 동작 명시 문서화 |

## 3. Subdomain ccTLD 대응

### 배경

현재 `SubdomainTenantExtractor`는 hostname을 `.`으로 split하여 `parts.length < 3`이면 `null`을 반환한다.
`tenant1.example.co.uk`처럼 multi-part TLD에서는 `parts.length === 4`이지만 `parts[0]`이 실제 서브도메인이 아닐 수 있다.

### 설계

**의존성**: `psl` (Public Suffix List) — optional dependency로 추가.

`psl`은 서버사이드에서 번들 크기 문제는 없지만, `SubdomainTenantExtractor`를 사용하지 않는 사용자에게도 설치를 강제하는 것은 부적절하다. optional dependency로 선언하고, 생성자에서 동적 import + 미설치 시 명확한 에러를 제공한다.

```typescript
// 사용법 — 기존 API 그대로
const extractor = new SubdomainTenantExtractor({
  excludeSubdomains: ['www'],
});

// tenant1.example.co.uk → tenant1
// www.example.co.uk → null (excluded)
// example.co.uk → null (no subdomain)
// tenant1.example.com → tenant1
// localhost → null (psl.parse error)
// 192.168.1.1 → null (psl.parse error)
```

**구현:**

```typescript
import type { ParsedDomain } from 'psl';

export class SubdomainTenantExtractor implements TenantExtractor {
  private readonly excludes: Set<string>;
  private psl: typeof import('psl') | null = null;

  constructor(options?: SubdomainExtractorOptions) {
    this.excludes = new Set(
      (options?.excludeSubdomains ?? ['www']).map((s) => s.toLowerCase()),
    );
    // 생성 시 psl 로드 시도
    try {
      this.psl = require('psl');
    } catch {
      throw new Error(
        'SubdomainTenantExtractor requires the "psl" package. Install it: npm install psl',
      );
    }
  }

  extract(request: Request): string | null {
    const hostname = request.hostname;
    const parsed = this.psl!.parse(hostname) as ParsedDomain;

    if ('error' in parsed || !parsed.subdomain) return null;

    // subdomain이 "a.b" 형태일 수 있으므로 첫 번째 파트만 추출
    const parts = parsed.subdomain.split('.');
    const subdomain = parts[0].toLowerCase();

    if (this.excludes.has(subdomain)) return null;
    return subdomain;
  }
}
```

**엣지 케이스:**
- `localhost` → `psl.parse()` 에러 반환 → `null`
- IP 주소 (`192.168.1.1`) → `psl.parse()` 에러 반환 → `null`
- `psl` 미설치 → 생성자에서 명확한 에러 throw

**하위 호환성:**
- `excludeSubdomains` 옵션 그대로 유지
- 기존 `.com`, `.org` 등 단순 TLD에서 동일 동작 (`psl`이 `parts[0]`과 동일 결과)
- `psl` 미설치 시 생성자 에러 — SubdomainTenantExtractor를 사용하지 않으면 영향 없음

**`@types/psl` 확인 결과:** `psl` 1.8+ 은 자체 TypeScript 선언을 내장한다. 별도 `@types/psl`은 불필요.

### 파일 변경

| 파일 | 변경 |
|------|------|
| `package.json` | `psl`을 `optionalDependencies`에 추가 |
| `src/extractors/subdomain.extractor.ts` | `psl.parse()` 기반 리팩터 + 동적 로드 |
| `test/subdomain.extractor.spec.ts` | ccTLD, localhost, IP 테스트 케이스 추가 |

## 4. Interactive Transaction 지원 (하이브리드)

### 배경

Prisma 확장의 `$allOperations`에서 배치 `$transaction([set_config, query])`를 사용하면, interactive transaction 내부에서는 `set_config`가 별도 커넥션에서 실행되어 RLS가 적용되지 않는다.

### 리서치 결과

- Prisma RLS 네이티브 지원 요청 (#12735): 161+ 추천, 4년째 오픈
- Prisma 공식 RLS 예제: "향후 버전에서 extension이 transaction 컨텍스트에 접근 가능" — 미이행
- 비공식 `__internalParams.transaction`으로 transaction 감지 가능하나 문서화되지 않음
- 사용자 반응: 래퍼 강제는 싫어함, 투명 처리 선호

### 설계: 하이브리드 접근

#### A. 안전한 기본값 — 독립 헬퍼 함수

`TenancyService`에 Prisma 의존성을 주입하지 않는다. 대신 독립적인 헬퍼 함수를 제공한다:

```typescript
// 사용법
import { tenancyTransaction } from '@nestarc/tenancy';

await tenancyTransaction(prisma, tenancyService, async (tx) => {
  const users = await tx.user.findMany();   // RLS 적용
  await tx.order.create({ data: { ... } }); // RLS 적용 + autoInject
});
```

내부 구현:
```typescript
export async function tenancyTransaction<T>(
  prisma: any,  // PrismaClient — any로 받아 Prisma 타입 의존 제거
  tenancyService: TenancyService,
  callback: (tx: any) => Promise<T>,
  options?: {
    timeout?: number;
    isolationLevel?: Prisma.TransactionIsolationLevel;
    dbSettingKey?: string;
  },
): Promise<T> {
  const tenantId = tenancyService.getCurrentTenantOrThrow();
  const settingKey = options?.dbSettingKey ?? 'app.current_tenant';

  return prisma.$transaction(async (tx: any) => {
    await tx.$executeRaw`SELECT set_config(${settingKey}, ${tenantId}, TRUE)`;
    return callback(tx);
  }, {
    timeout: options?.timeout,
    isolationLevel: options?.isolationLevel,
  });
}
```

**설계 근거:**
- `TenancyService`는 현재 Prisma 의존이 전혀 없다 — 이 분리를 유지
- 독립 함수이므로 `PrismaClient` 타입을 import하지 않아도 됨 (`any`로 수용)
- `Prisma.TransactionIsolationLevel`은 타입 레벨에서만 사용 (런타임 의존 아님)
- `TransactionClient` 타입을 사용자에게 노출하지 않음 — 콜백의 `tx`는 `any`

#### B. Experimental 투명 모드 — opt-in

```typescript
createPrismaTenancyExtension(tenancyService, {
  experimentalTransactionSupport: true,
});

// 이후 일반 $transaction도 자동 동작
await prisma.$transaction(async (tx) => {
  await tx.user.findMany(); // RLS 자동 적용
});
```

내부 구현: `$allOperations`에서 `(params as any).__internalParams?.transaction`을 검사하여, interactive transaction 내부(`kind === 'itx'`)이면 해당 transaction 커넥션에 `set_config`를 주입.

**런타임 안전장치:**
- `experimentalTransactionSupport: true` 설정 시, 첫 번째 쿼리에서 `__internalParams`의 존재를 확인
- 내부 API가 감지되지 않으면 `console.warn`으로 경고 + 기존 배치 트랜잭션 폴백
- JSDoc, README에 Prisma 호환 버전 명시 (테스트 통과 확인된 버전)

**주의사항 (JSDoc + README에 문서화):**
- `experimentalTransactionSupport`는 Prisma 비공식 내부 API에 의존
- Prisma 버전 업그레이드 시 동작이 깨질 수 있음 — 런타임 경고로 감지
- Prisma가 공식 transaction context API를 제공하면 내부 구현 교체 예정

| 파일 | 변경 |
|------|------|
| `src/prisma/tenancy-transaction.ts` | `tenancyTransaction()` 독립 헬퍼 함수 |
| `src/prisma/prisma-tenancy.extension.ts` | `experimentalTransactionSupport` 옵션 + 내부 감지 + 런타임 경고 |

## 5. CLI 도구 (`npx @nestarc/tenancy init`)

### 배경

새 사용자의 도입 비용을 최소화한다. RLS 정책 작성, Prisma 스키마 수정, 모듈 등록 코드를 수동으로 하면 실수하기 쉽다.

### 설계

```bash
npx @nestarc/tenancy init
```

**`bin` 필드:** scoped 패키지에서 `npx @nestarc/tenancy`로 실행하려면 bin 키를 패키지명과 일치시켜야 한다:
```json
{
  "bin": {
    "tenancy": "./dist/cli/index.js"
  }
}
```

**대화형 프롬프트:**

```
? Tenant extraction strategy
  ❯ Header (X-Tenant-Id)
    Subdomain (tenant1.app.com)
    JWT Claim
    Path Parameter
    Composite (multiple strategies)

? Tenant ID format
  ❯ UUID
    Custom (provide regex)

? Database setting key (app.current_tenant)

? Enable auto-inject tenant ID on writes? (Y/n)

? Shared models (comma-separated, e.g., Country,Currency):
```

**생성 파일:**

1. **`tenancy-setup.sql`** — RLS 정책 + app_user 롤 생성 스크립트
   ```sql
   -- Generated by @nestarc/tenancy CLI
   CREATE ROLE app_user WITH LOGIN PASSWORD 'changeme';

   ALTER TABLE "User" ENABLE ROW LEVEL SECURITY;
   CREATE POLICY tenant_isolation_User ON "User"
     USING (tenant_id = current_setting('app.current_tenant', true)::text);
   GRANT SELECT, INSERT, UPDATE, DELETE ON "User" TO app_user;
   ```

2. **`tenancy.module-setup.ts`** — 모듈 등록 코드 예제
   ```typescript
   // Add to your AppModule imports:
   TenancyModule.forRoot({
     tenantExtractor: 'X-Tenant-Id',
     validateTenantId: (id) => /^[0-9a-f-]{36}$/i.test(id),
   })
   ```

3. **콘솔 가이드** — schema.prisma에 tenant_id 추가 안내, DATABASE_URL 설정 안내

**Prisma 스키마 파싱:**
- 정규식: `/^model\s+(\w+)\s*\{/gm` 으로 모델명 추출
- `@@map("table_name")`이 있으면 해당 테이블명을 SQL에 사용
- `@@map` 파싱 정규식: `/@@map\("([^"]+)"\)/`
- `@@map`이 없으면 Prisma 모델명을 그대로 테이블명으로 사용
- enum, type, view 는 `model` 키워드가 아니므로 자동 제외

**파일 충돌 처리:**
- 대상 파일이 이미 존재하면 프롬프트: `tenancy-setup.sql already exists. Overwrite? (y/N)`
- 기본값 No — 사용자 커스터마이징 보호

**의존성:** `prompts`는 `optionalDependencies`에 추가. CLI를 사용하지 않는 사용자에게 설치를 강제하지 않는다. CLI 실행 시 `require('prompts')`가 실패하면 "Install prompts: npm install prompts" 에러를 표시한다.

### 파일 변경

| 파일 | 변경 |
|------|------|
| `src/cli/index.ts` | CLI 진입점 (init 서브커맨드 라우팅) |
| `src/cli/init.ts` | `init` 명령 로직 (프롬프트 + 파일 생성) |
| `src/cli/templates/setup.sql.ts` | SQL 템플릿 생성 함수 |
| `src/cli/templates/module-setup.ts.ts` | 모듈 코드 템플릿 생성 함수 |
| `src/cli/prisma-schema-parser.ts` | schema.prisma 모델 + @@map 파싱 |
| `package.json` | `bin` 필드 + `prompts`를 optionalDependencies |
| `tsconfig.build.json` | `src/cli/` include 확인 |
| `test/cli/init.spec.ts` | CLI 유닛 테스트 (파일 생성, 충돌 처리) |
| `test/cli/prisma-schema-parser.spec.ts` | 스키마 파싱 테스트 (@@map 포함) |

## 7. 하위 호환성

모든 변경은 **하위 호환**:

- `withoutTenant()`: 새 메서드 추가, 기존 API 변경 없음
- `@BypassTenancy()`: 기존 동작 그대로 (README 문서화만 추가)
- `SubdomainTenantExtractor`: 기존 API 동일, 내부 파싱만 `psl`로 교체. 단, `psl` 미설치 시 생성자 에러 — **breaking change**이므로 마이그레이션 가이드 필요
- `tenancyTransaction()`: 새 함수 추가 (기존 API 변경 없음)
- `experimentalTransactionSupport`: opt-in (기본값 `false`)
- CLI: 새 명령 추가 (기존 라이브러리 동작 무관)

**SubdomainTenantExtractor breaking change 대응:**
- CHANGELOG에 migration guide 포함
- README에 `npm install psl` 안내 추가
- `psl` 미설치 시 에러 메시지에 설치 명령 포함

## 8. 테스트 전략

| 기능 | 유닛 테스트 | E2E 테스트 |
|------|-----------|-----------|
