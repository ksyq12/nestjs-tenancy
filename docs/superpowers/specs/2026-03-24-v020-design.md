# @nestarc/tenancy v0.2.0 Design Spec

> Date: 2026-03-24
> Status: Approved
> Approach: Incremental extension (backward-compatible with v0.1.0)

---

## Overview

v0.2.0 adds three features to the existing `@nestarc/tenancy` module:

1. **Multiple tenant extraction strategies** ‚Äî subdomain, JWT claim, URL path, and composite (fallback chain)
2. **Tenant lifecycle hooks** ‚Äî `onTenantResolved` and `onTenantNotFound` callbacks
3. **Prisma extension enhancements** ‚Äî `autoInjectTenantId`, `tenantIdField`, `sharedModels`

All changes are additive. Existing v0.1.0 configurations continue to work without modification.

---

## 1. Multiple Extraction Strategies

### 1.1 Interface ‚Äî No Change

All new extractors implement the existing `TenantExtractor` interface:

```typescript
export interface TenantExtractor {
  extract(request: Request): string | null | Promise<string | null>;
}
```

### 1.2 New Files

```
src/extractors/
‚îú‚îÄ‚îÄ header.extractor.ts          # Existing (unchanged)
‚îú‚îÄ‚îÄ subdomain.extractor.ts       # New
‚îú‚îÄ‚îÄ jwt-claim.extractor.ts       # New
‚îú‚îÄ‚îÄ path.extractor.ts            # New
‚îî‚îÄ‚îÄ composite.extractor.ts       # New
```

### 1.3 SubdomainTenantExtractor

```typescript
new SubdomainTenantExtractor(options?: { excludeSubdomains?: string[] })
```

- Extracts the first label from `req.hostname` (e.g., `tenant1.app.com` ‚Üí `tenant1`)
- Returns `null` if no subdomain or subdomain is in exclude list
- Default `excludeSubdomains`: `['www']`
- Does NOT split by port ‚Äî `req.hostname` already excludes port in Express

### 1.4 JwtClaimTenantExtractor

```typescript
new JwtClaimTenantExtractor(options: { claimKey: string; headerName?: string })
```

- Reads `Authorization: Bearer <token>` header (or custom `headerName`)
- Base64url-decodes the JWT payload using `Buffer.from(str, 'base64url')` (Node 16+). JWT payloads use Base64url encoding (RFC 7515), NOT standard Base64. No verification ‚Äî auth is handled by separate guards.
- Extracts `payload[claimKey]` as tenant ID
- Returns `null` if header missing, malformed, or claim absent
- Zero external dependencies (no `jsonwebtoken` package)

### 1.5 PathTenantExtractor

```typescript
new PathTenantExtractor(options: { pattern: string; paramName: string })
```

- `pattern`: URL path template with `:paramName` placeholder (e.g., `/api/tenants/:tenantId`)
- Splits both pattern and `req.path` by `/` and matches segments positionally
- **Prefix match**: the request path may have additional trailing segments beyond the pattern (e.g., pattern `/api/tenants/:tenantId` matches `/api/tenants/abc123/users/profile`). Only the segments covered by the pattern must match; extra trailing segments are ignored.
- Extracts the segment at the `:paramName` position
- Returns `null` if the request path has fewer segments than the pattern, or if non-parameter segments don't match
- No external dependencies (no `path-to-regexp`)

### 1.6 CompositeTenantExtractor

```typescript
new CompositeTenantExtractor(extractors: TenantExtractor[])
```

- Iterates extractors in array order
- Returns the first non-null result
- Returns `null` if all extractors return null
- Supports async extractors (awaits each in sequence)

### 1.7 Backward Compatibility

`TenancyModuleOptions.tenantExtractor` type remains `string | TenantExtractor`. No changes needed:

```typescript
// v0.1.0 usage ‚Äî still works
TenancyModule.forRoot({ tenantExtractor: 'X-Tenant-Id' })

// v0.2.0 usage ‚Äî new extractors
TenancyModule.forRoot({
  tenantExtractor: new CompositeTenantExtractor([
    new HeaderTenantExtractor('X-Tenant-Id'),
    new SubdomainTenantExtractor(),
  ]),
})
```

### 1.8 Barrel Exports

All extractors are exported from `src/index.ts`.

## 2. Tenant Lifecycle Hooks

### 2.1 Interface Extension

Two optional fields added to `TenancyModuleOptions`:

```typescript
export interface TenancyModuleOptions {
  tenantExtractor: string | TenantExtractor;
  dbSettingKey?: string;
  validateTenantId?: (tenantId: string) => boolean | Promise<boolean>;

  // v0.2.0
  onTenantResolved?: (tenantId: string, request: Request) => void | Promise<void>;
  onTenantNotFound?: (request: Request) => void | Promise<void>;
}
```

### 2.2 Call Site: TenantMiddleware

```typescript
async use(req: Request, _res: Response, next: NextFunction): Promise<void> {
  const tenantId = await this.extractor.extract(req);

  if (!tenantId) {
    await this.options.onTenantNotFound?.(req);   // Hook
    next();
    return;
  }

  const isValid = await this.validate(tenantId);
  if (!isValid) {
    throw new BadRequestException('Invalid tenant ID format');
  }

  this.context.run(tenantId, async () => {
    await this.options.onTenantResolved?.(tenantId, req);   // Hook (inside context)
    next();
  });
}
```

### 2.3 Design Decisions

- **Return type**: `void | Promise<void>` ‚Äî hooks are notifications, not flow control
- **Timing**: `onTenantResolved` fires INSIDE `context.run()`, AFTER validation. This ensures that any service called from within the hook can use `TenancyService.getCurrentTenant()` and get the correct tenant ID. The hook also receives `tenantId` as a direct argument for convenience.
- **Error propagation**: If a hook throws, the exception propagates through NestJS exception filters. This is intentional ‚Äî hooks can reject requests by throwing.
- **Async support**: Hooks are awaited, enabling async operations (DB logging, external API calls).

### 2.3.1 Known Limitation: `onTenantNotFound` and `@BypassTenancy`

The middleware runs on ALL routes before the guard checks `@BypassTenancy()`. This means `onTenantNotFound` will fire on `@BypassTenancy()` routes when no tenant header is present. This is expected ‚Äî the middleware cannot access route metadata. If this causes noise in logging hooks, users should filter by `req.path` within their hook implementation.

### 2.4 Middleware Constructor Changes

The middleware constructor now stores the full options object (currently it only stores `extractor` and `validate`):

```typescript
constructor(
  @Inject(TENANCY_MODULE_OPTIONS) private readonly options: TenancyModuleOptions,
  private readonly context: TenancyContext,
) {
  this.extractor = typeof options.tenantExtractor === 'string'
    ? new HeaderTenantExtractor(options.tenantExtractor)
    : options.tenantExtractor;
  this.validate = options.validateTenantId ?? ((id: string) => UUID_REGEX.test(id));
}
```

The `options` reference is stored as a private field to access hooks.

## 3. Prisma Extension Enhancements

### 3.1 Extended Options Interface

```typescript
export interface PrismaTenancyExtensionOptions {
  dbSettingKey?: string;

  // v0.2.0
  autoInjectTenantId?: boolean;   // Default: false (opt-in)
  tenantIdField?: string;         // Default: 'tenant_id'
  sharedModels?: string[];        // Models that skip set_config (e.g., ['Country', 'Currency'])
                                  // WARNING: Model names are case-sensitive and must match Prisma schema exactly (PascalCase).
}
```

### 3.2 sharedModels ‚Äî RLS Bypass by Model

In `$allOperations`, check `model` against `sharedModels` array:

```typescript
async $allOperations({ model, operation, args, query }) {
  const tenantId = tenancyService.getCurrentTenant();

  if (!tenantId || sharedModels.includes(model)) {
    return query(args);
  }

  // Existing: set_config + query transaction
  const [, result] = await baseClient.$transaction([
    baseClient.$executeRaw`SELECT set_config(${settingKey}, ${tenantId}, TRUE)`,
    query(args),
  ]);
  return result;
}
```

### 3.3 autoInjectTenantId ‚Äî Automatic Column Injection

When `autoInjectTenantId: true`, the extension injects `tenantIdField` into write operations:

| Operation | Injection Behavior |
|-----------|--------------------|
| `create` | `args.data = { ...args.data, [field]: tenantId }` (spread copy) |
| `createMany` | `args.data = args.data.map(d => ({ ...d, [field]: tenantId }))` (spread copy per item) |
| `upsert` | `args.create = { ...args.create, [field]: tenantId }` (update left unchanged) |
| `update` | No injection ‚Äî RLS already prevents cross-tenant updates |
| `updateMany` | No injection ‚Äî same reason |
| `delete` / `deleteMany` / reads | No injection ‚Äî not applicable |

**Spread order**: `{ ...args.data, [field]: tenantId }` ‚Äî library value wins. If the user explicitly provides a different tenant_id, the library overwrites it for safety. This prevents accidental cross-tenant writes.

**Interaction with sharedModels**: Models in `sharedModels` skip both set_config AND auto-injection.

### 3.4 $allOperations Signature

The current implementation doesn't destructure `model` or `operation`. Update the signature:

```typescript
async $allOperations({
  model,
  operation,
  args,
  query,
}: {
  model: string;
  operation: string;
  args: any;
  query: (args: any) => Promise<any>;
}) {
  // ...
}
```

### 3.5 Full Extension Flow

```
Prisma query intercepted by Extension
  ‚Üí Check: tenant context exists?
    ‚Üí No: run query directly (no set_config, no injection)
  ‚Üí Check: model in sharedModels?
    ‚Üí Yes: run query directly
  ‚Üí Check: autoInjectTenantId enabled?
    ‚Üí Yes + write operation: inject tenantIdField into args.data
  ‚Üí Execute: $transaction([set_config, query])
  ‚Üí Return result
```

## 6. Migration Guide (v0.1.0 ‚Üí v0.2.0)

**No breaking changes.** All new features are opt-in:

```typescript
// v0.1.0 config ‚Äî still works identically in v0.2.0
TenancyModule.forRoot({
  tenantExtractor: 'X-Tenant-Id',
})

// v0.2.0 ‚Äî opt into new features
TenancyModule.forRoot({
  tenantExtractor: new CompositeTenantExtractor([
    new HeaderTenantExtractor('X-Tenant-Id'),
    new SubdomainTenantExtractor(),
  ]),
  onTenantResolved: (tenantId, req) => {
    console.log(`Tenant ${tenantId} resolved for ${req.path}`);
  },
})

// Prisma extension ‚Äî opt into new features
createPrismaTenancyExtension(tenancyService, {
  autoInjectTenantId: true,
  tenantIdField: 'organization_id',
  sharedModels: ['Country', 'Currency'],
})

// v0.2.0 ‚Äî forRootAsync with hooks
TenancyModule.forRootAsync({
  imports: [ConfigModule],
  useFactory: (config: ConfigService) => ({
    tenantExtractor: config.get('TENANT_HEADER') ?? 'X-Tenant-Id',
    onTenantResolved: (tenantId, req) => {
      console.log(`Tenant ${tenantId} resolved for ${req.path}`);
    },
  }),
  inject: [ConfigService],
})
```
